# Техническое задание для ИИ-агента разработчика

## Проект: модульная система авторского переписывания текста с опорой на референсный стиль

---

## 1. Назначение системы

Необходимо разработать **Python-friendly модульную систему авторского переписывания текста**, которая подключается к любому внешнему проекту по API и выполняет:

* анализ пользовательских библиотек референсных текстов;
* построение стилевого профиля на основе этих библиотек;
* переписывание входного текста под выбранный стиль;
* сохранение смыслового ядра текста;
* оценку качества результата;
* предоставление результатов через API в формате, удобном для интеграции в сторонние продукты.

Система должна быть спроектирована как **независимый сервисный модуль**, а не как жёстко связанное приложение. Она должна уметь работать как отдельный backend-сервис, который можно подключить к CMS, редактору, контентной платформе, агентной системе, workflow-движку, CRM или любому другому продукту по HTTP API.

Архитектурно система должна соответствовать концепции **reference-driven rewriting system**, где ключевой источник целевого стиля — это библиотека референсных текстов, а не абстрактное понятие “естественного текста” .

---

# 2. Бизнес-цель системы

Система должна решать следующую задачу:

**превращать исходный текст в текст, который сохраняет смысл и звучит так, как будто он естественно относится к выбранному корпусу референсных текстов.**

Система не должна ограничиваться поверхностным перефразированием. Она должна:

* сохранять факты, имена, цифры, причинно-следственные связи и терминологию;
* переносить текст в целевой стиль;
* делать это управляемо и прозрачно;
* позволять внешней системе или оператору контролировать степень вмешательства;
* обеспечивать качество результата через автоматическую и полуавтоматическую оценку.

---

# 3. Общие архитектурные требования

## 3.1. Формат поставки

Результат разработки должен быть оформлен как:

* backend-модуль / сервис;
* API-first система;
* разворачиваемый Python-сервис;
* независимый компонент, подключаемый к внешним системам.

## 3.2. Технологические требования

Система должна быть максимально **Python friendly** и использовать стек, естественный для Python-экосистемы.

### Обязательные технологические ориентиры

* основной язык: **Python**
* API-слой: **FastAPI**
* хранилище: **PostgreSQL**
* быстрый кэш / очереди / промежуточное состояние: **Redis**
* асинхронные задачи: **Celery** или аналогичный Python-friendly инструмент
* NLP/analysis слой: Python-стек
* интеграция со сторонними LLM: через **OpenAI API** и расширяемый интерфейс для других провайдеров
* система должна быть контейнеризуемой
* все внешние зависимости должны быть отделены интерфейсами и адаптерами

## 3.3. Документация

Все документы проекта должны считаться доступными в каталоге:

`docs/`

При проектировании и реализации агент должен ссылаться на документы проекта именно через путь `docs/...`.

# 4. Обязательные проектные принципы

## 4.1. Система должна разделять смысл и стиль

Архитектура должна быть выстроена так, чтобы:

* **смысл** рассматривался как инвариантная сущность;
* **стиль** рассматривался как управляемая переменная.

Это означает, что все ключевые модули должны различать:

* смысловое содержимое текста;
* выразительные свойства текста.

## 4.2. Система должна быть reference-driven

В основе системы должны лежать **Style Libraries** — библиотеки референсных текстов, из которых система строит профиль стиля.

## 4.3. Система должна быть guided, а не blind

Переписывание должно быть устроено не по схеме “переписали, потом проверили”, а по схеме:

* генерация нескольких вариантов;
* оценка вариантов внешними модулями;
* выбор лучшего или нескольких лучших;
* финальная полировка;
* оценка.

То есть rewrite engine должен работать совместно с **guidance-логикой**. Это соответствует ранее зафиксированной архитектуре проекта и идее guided rewriting, а не слепого rewrite с одной финальной проверкой .

## 4.4. Система должна быть модульной

Каждая крупная функция должна быть реализована отдельным модулем или сервисным слоем с чётким контрактом.

## 4.5. Система должна быть пригодна для human-in-the-loop сценариев

Даже если первичная интеграция будет машинной, система должна возвращать достаточно данных для:

* визуального diff;
* объяснения решений;
* выбора варианта;
* ручной редакторской корректировки.

# 6. Основные сущности доменной модели

Агент-разработчик должен заложить следующую доменную модель.

## 6.1. Style Library

Сущность, представляющая библиотеку референсных текстов одного типа.

Содержит:

* имя библиотеки;
* описание;
* категорию;
* язык;
* область применения;
* владельца;
* проект;
* статус;
* версию;
* список текстов-образцов;
* рассчитанный стиль-профиль;
* уровень качества библиотеки.

## 6.2. Style Sample

Отдельный текст внутри библиотеки.

Содержит:

* текст;
* заголовок;
* источник;
* тип контента;
* предполагаемый стиль;
* автор;
* язык;
* качество;
* служебные метаданные;
* результат анализа.

## 6.3. Style Profile

Вычисленный стилевой профиль библиотеки.

Содержит:

* формальность;
* среднюю длину предложений;
* разброс длины;
* ритм;
* типичную плотность абзацев;
* характер переходов;
* степень образности;
* частотные маркеры;
* нежелательные маркеры;
* типовые синтаксические паттерны;
* профиль композиции;
* служебные признаки для style guidance.

## 6.4. Semantic Contract

Сущность, описывающая смысловые ограничения rewrite-задачи.

Содержит:

* защищённые сущности;
* защищённые факты;
* числовые ограничения;
* причинно-следственные опоры;
* список терминов с обязательным сохранением;
* список чувствительных сегментов;
* allowed rewrite level;
* режим strict/balanced/loose.

## 6.5. Rewrite Task

Конкретная задача переписывания.

Содержит:

* исходный текст;
* выбранную библиотеку стиля;
* режим rewrite;
* режим semantic contract;
* входные ограничения;
* идентификатор проекта;
* идентификатор пользователя / клиента;
* служебное состояние выполнения.

## 6.6. Rewrite Variant

Один вариант результата.

Содержит:

* итоговый текст;
* промежуточные показатели;
* метрики;
* оценку style match;
* оценку semantic preservation;
* объяснение изменений;
* статус пригодности.

## 6.7. Evaluation Report

Итоговый отчёт оценки качества.

Содержит:

* абсолютные метрики;
* judge-оценки;
* pairwise-оценки;
* нарушения semantic contract;
* предупреждения;
* рекомендации.

## 6.8. Style Preset

Сохранённая конфигурация работы системы.

Содержит:

* Style Library;
* rewrite mode;
* semantic contract mode;
* набор ограничений;
* уровень вмешательства;
* активные эвристики.

# 7. Описание модулей

## 7.1. Reference Library Manager

### Назначение

Управление библиотеками референсных текстов.

### Обязанности

* создание новой библиотеки;
* добавление текстов;
* импорт текстов пакетами;
* обновление библиотеки;
* связывание библиотеки с проектом;
* версионирование;
* архивирование;
* удаление / деактивация;
* переиндексация.

### Требования

* библиотека должна поддерживать тексты одного языка;
* библиотека должна иметь категорию;
* библиотека должна поддерживать служебные теги;
* должна быть возможность хранить несколько библиотек для одного проекта;
* должна быть возможность создавать как “личные”, так и “проектные” библиотеки.

### Важное правило

Библиотека не должна считаться валидной для профилирования, пока не пройдет минимальную проверку качества и объёма.

---

## 7.2. Corpus Quality Tiering

### Назначение

Оценка качества библиотеки и отдельных sample’ов.

### Почему это обязательно

Корпус из смешанных или слабых текстов создаёт плохой стилевой профиль. Следовательно, система должна определять качество корпусов, а не принимать все тексты как равнозначные.

### Требуемые уровни

* **L1** — сильный, чистый, консистентный стиль
* **L2** — рабочий, но неоднородный стиль
* **L3** — шумный, слабый или конфликтный материал

### Что модуль должен уметь

* оценивать каждый загруженный sample;
* оценивать библиотеку целиком;
* определять, можно ли использовать sample в основном профиле;
* определять, нужно ли исключить sample;
* формировать рекомендации по очистке библиотеки.

### Практическая логика

* в расчёт основного Style Profile по умолчанию должны входить только L1 и, при необходимости, L2 sample’ы;
* L3 sample’ы должны либо исключаться, либо учитываться только в исследовательском/вспомогательном контуре.

---

## 7.3. Style Profile Engine

### Назначение

Построение машиночитаемого профиля стиля по библиотеке.

### Что должен анализировать

* длину предложений;
* распределение длин;
* длину абзацев;
* ритм;
* уровень формальности;
* плотность оценочной лексики;
* типичные переходы;
* уровень образности;
* частоту определённых вводных конструкций;
* характер синтаксиса;
* композиционные привычки;
* предпочтительные и нежелательные лексические маркеры.

### Выходные данные

Модуль должен выдавать:

1. **описательный профиль стиля**;
2. **сигналы для guidance-модуля**;
3. **признаки для style match scoring**;
4. **сигналы отклонения от стиля**.

### Важное требование

Этот модуль не должен ограничиваться статистическим summary. Он должен строить именно **Style DNA**, пригодный для дальнейшего управления rewrite-задачей.

---

## 7.4. Input Analyzer

### Назначение

Анализировать входной текст до переписывания.

### Обязанности

* определять жанровые характеристики;
* оценивать текущую формальность;
* выявлять шаблонность;
* оценивать плотность фактов;
* оценивать насыщенность конкретикой;
* находить повторяемые паттерны;
* строить risk map;
* рекомендовать глубину rewrite.

### Выход

* входной технический профиль;
* карта участков с высоким риском потери смысла;
* карта участков с высокой стилевой гибкостью.

---

## 7.5. Semantic Contract Builder

### Назначение

Построить контракт, защищающий смысл текста.

### Обязанности

* извлечь ключевые сущности;
* извлечь числа, даты, имена, термины;
* определить чувствительные фрагменты;
* определить обязательные для сохранения утверждения;
* определить допустимую глубину rewrite.

### Режимы

* **Strict**
* **Balanced**
* **Loose**

### Внутренняя карта важности

Модуль должен разделять содержимое текста на:

* must-preserve;
* high-risk;
* replaceable;
* style-flex.

### Дополнительное требование

Semantic Contract Builder должен уметь возвращать объяснимые ограничения, пригодные для внешнего API и пользовательского интерфейса.

---

## 7.6. Guided Rewrite Engine

### Назначение

Выполнять управляемое переписывание текста.

### Требования к реализации

* базироваться на encoder-decoder логике rewrite;
* быть совместимым с внешними LLM через абстракции;
* поддерживать несколько режимов генерации;
* уметь работать по сегментам;
* поддерживать генерацию нескольких вариантов;
* не быть привязанным к одному поставщику LLM.

### Обязательные режимы

* Conservative
* Balanced
* Expressive

### Обязательные требования

* учёт semantic contract;
* учёт style profile;
* возможность отдачи нескольких кандидатов на следующий этап;
* отсутствие жёсткой зависимости от OpenAI API;
* возможность использовать OpenAI API как один из провайдеров.

---

## 7.7. LLM Provider Abstraction Layer

### Назначение

Сделать всю LLM-часть системы заменяемой.

### Система должна поддерживать

* внешний OpenAI API;
* возможность подключения других LLM-провайдеров;
* локальные модели в будущем;
* единый внутренний интерфейс вызова модели.

### Обязательные задачи абстракции

* нормализация запросов;
* нормализация ответов;
* повторные запросы;
* rate limit handling;
* таймауты;
* трассировка;
* логирование;
* политика стоимости и квот.

### Важно

OpenAI API должен учитываться как штатный провайдер, но реализация не должна становиться OpenAI-only.

---

## 7.8. Style Guidance Engine

### Назначение

Направлять rewrite-процесс к целевому стилю.

### Что должен делать

* получать несколько candidate rewrites;
* оценивать их по совпадению со стилем;
* учитывать semantic safety;
* учитывать ритм;
* учитывать композицию;
* штрафовать варианты, уходящие в плохую гладкость, штампы или потерю авторского профиля;
* отдавать ranked list кандидатов.

### Важно

Этот модуль должен работать **не только после генерации**, но и как рабочий управляющий слой в rewrite pipeline.

---

## 7.9. Holistic Lexical Substitution Ranker

### Назначение

Управлять локальными заменами слов и фраз с учетом всего предложения.

### Основные требования

* оценивать замену не только по слову, но и по влиянию на весь сегмент;
* учитывать контекст;
* учитывать style profile;
* учитывать semantic contract;
* учитывать phrase-level coherence.

### Режимы

* Fast mode
* Precision mode

### Где должен применяться

* точечная пост-обработка;
* финальный polishing;
* интерактивные редакторские предложения;
* локальные rewrite-операции.

---

## 7.10. Rule-Based Structural Polishing

### Назначение

Финально стабилизировать и улучшать структуру текста.

### Что должен делать

* убирать штампы;
* заменять нежелательные вводные;
* выравнивать или, наоборот, оживлять ритм;
* разбивать перегруженные предложения;
* объединять слишком рубленые сегменты;
* убирать канцелярит;
* исправлять избыточную симметрию;
* учитывать стиль выбранной библиотеки.

### Внутренний constraint layer

Этот модуль обязан учитывать:

* maximum token rewrite ratio;
* maximum sentence drift;
* maximum paragraph drift;
* protected phrase spans;
* POS-aware rewrite rules.

---

## 7.11. Grammar Layer

### Назначение

Финальная языковая полировка текста.

### Что должен делать

* грамматическая проверка;
* пунктуация;
* согласование;
* типографика;
* унификация терминологии;
* удаление артефактов после rewrite.

### Важное ограничение

Не должен убивать индивидуальность текста чрезмерной стерилизацией.

---

## 7.12. Evaluation Engine

### Назначение

Проводить полноценную оценку результата.

### Должен иметь три режима оценки

#### 1. Absolute Metrics

* semantic similarity;
* entity preservation;
* fact preservation;
* style match;
* readability;
* lexical diversity;
* repetition score;
* cliché score.

#### 2. Judge Evaluation

* оценка LLM-судьёй;
* проверка естественности;
* проверка сохранения смысла;
* проверка стилевого соответствия.

#### 3. Pairwise Comparison

* сравнение вариантов друг с другом;
* выбор лучшего варианта по смыслу, стилю, ясности и естественности.

### Результат работы модуля

* структурированный evaluation report;
* итоговый composite score;
* объяснение слабых мест;
* предупреждения.

---

## 7.13. Human-in-the-loop Support Layer

### Назначение

Обеспечить совместимость с интерфейсами, в которых человек выбирает или редактирует результат.

### Обязанности

* возвращать diff;
* возвращать список изменений;
* отмечать risky segments;
* отдавать альтернативные варианты;
* возвращать score breakdown;
* поддерживать сравнение A/B/C.

### Важно

Даже если на первом этапе фронтенд не разрабатывается, backend должен быть готов к human-in-the-loop интеграции.

# 8. Требования к API

## 8.1. Общие требования

API должно быть:

* RESTful;
* стабильным;
* versioned;
* документированным;
* пригодным для внешней интеграции;
* пригодным для асинхронной обработки долгих задач.

## 8.2. Обязательные группы API

### Libraries API

* создать библиотеку;
* получить библиотеку;
* обновить библиотеку;
* удалить/архивировать библиотеку;
* добавить sample;
* удалить sample;
* пересчитать профиль;
* получить quality tiering.

### Profiles API

* получить стиль-профиль;
* получить style DNA;
* получить style warnings;
* получить library diagnostics.

### Rewrite API

* создать rewrite task;
* запустить rewrite;
* получить варианты;
* получить статус;
* получить результаты;
* получить оценку.

### Evaluation API

* оценить готовый текст;
* сравнить 2 варианта;
* сравнить несколько вариантов;
* проверить semantic preservation;
* проверить style match.

### Presets API

* сохранить preset;
* получить preset;
* применить preset.

### Admin / Tenant API

* привязать библиотеку к проекту;
* управлять квотами;
* управлять LLM-провайдерами;
* управлять настройками доступа.

---

# 9. Интеграционный сценарий

Система должна уметь работать так:

1. Внешний проект создаёт Style Library.
2. Загружает 10–50 референсных текстов.
3. Система рассчитывает quality tiering и style profile.
4. Внешний проект отправляет исходный текст на rewrite.
5. Система строит semantic contract.
6. Система генерирует несколько rewrite-вариантов.
7. Guidance + Evaluation выбирают лучшие или возвращают все.
8. Внешний проект получает:

   * итоговые тексты;
   * scores;
   * diff / explanation data;
   * warnings.

---

# 10. Требования к объёму корпусов стиля

Агент-разработчик должен учитывать в логике системы:

* библиотека из 5–7 текстов может использоваться только как слабый старт;
* библиотека из 10–15 текстов — минимально рабочая;
* библиотека из 20–30 текстов — хороший рабочий объём;
* библиотека из 30–50 текстов — сильный корпус при условии однородности.

Система должна диагностировать не только объём, но и **качество и однородность** библиотеки.

---

# 11. Логика работы с авторами и смешанными корпусами

Система должна поддерживать два режима библиотек:

## 11.1. Single-voice library

Для конкретного авторского голоса.

## 11.2. Editorial-style library

Для стилевого профиля жанра / рубрики / медиа / бренда.

Система должна различать:

* библиотеку “одного голоса”;
* библиотеку “общего редакционного стиля”.

Если внутри одной библиотеки стиль конфликтует, система должна:

* либо сигнализировать о неоднородности;
* либо рекомендовать разбить библиотеку на несколько.

# 12. Нефункциональные требования

## 12.1. Производительность

* должна быть поддержка асинхронного режима;
* тяжелые задачи должны уходить в background jobs;
* API должно быстро возвращать task id для долгих операций.

## 12.2. Надёжность

* идемпотентность на ключевых операциях;
* retry-safe обработка;
* логирование;
* трассировка ошибок;
* устойчивость к сбоям LLM-провайдера.

## 12.3. Расширяемость

* возможность подключить новые провайдеры LLM;
* возможность добавить новые evaluation backends;
* возможность добавить новые типы style profiles;
* возможность расширить систему до multi-tenant режима.

## 12.4. Безопасность

* изоляция tenant/project данных;
* контроль доступа к библиотекам;
* управление API-ключами внешних провайдеров;
* защита от несанкционированного доступа к чужим corpus/profile данным.

## 12.5. Наблюдаемость

* метрики запросов;
* метрики ошибок;
* метрики latency;
* стоимость вызовов LLM;
* доля успешных rewrite-задач;
* распределение quality scores;
* диагностика library quality.

# 17. Что не должно быть упущено разработчиком

Агент обязан учесть:

1. что система — **API-модуль**, а не привязанный UI-продукт;
2. что OpenAI API должен быть встроен как внешний LLM-провайдер;
3. что архитектура должна оставаться provider-agnostic;
4. что смысл должен быть защищён через semantic contract;
5. что rewrite должен быть guided;
6. что style profile должен быть пригоден не только для хранения, но и для управления rewrite;
7. что библиотеки стиля должны проходить quality tiering;
8. что оценка результата должна быть многослойной;
9. что всё должно быть Python-friendly;
10. что кодовые детали агент выбирает сам, но архитектурные принципы менять нельзя.
