# Техническое задание для ИИ-агента разработчика

## 1. Назначение системы

Необходимо разработать **Python-friendly модульную систему авторского переписывания текста**, которая подключается к любому внешнему проекту по API и выполняет:

* анализ пользовательских библиотек референсных текстов;
* построение стилевого профиля на основе этих библиотек;
* переписывание входного текста под выбранный стиль;
* сохранение смыслового ядра текста;
* оценку качества результата;
* предоставление результатов через API в формате, удобном для интеграции в сторонние продукты.

Система должна быть спроектирована как **независимый сервисный модуль**, а не как жёстко связанное приложение. Она должна уметь работать как отдельный backend-сервис, который можно подключить к CMS, редактору, контентной платформе, агентной системе, workflow-движку, CRM или любому другому продукту по HTTP API.

Архитектурно система должна соответствовать концепции **reference-driven rewriting system**, где ключевой источник целевого стиля — это библиотека референсных текстов, а не абстрактное понятие "естественного текста".

---

# 2. Бизнес-цель системы

Система должна решать следующую задачу:

**превращать исходный текст в текст, который сохраняет смысл и звучит так, как будто он естественно относится к выбранному корпусу референсных текстов.**

Система не должна ограничиваться поверхностным перефразированием. Она должна:

* сохранять факты, имена, цифры, причинно-следственные связи и терминологию;
* переносить текст в целевой стиль;
* делать это управляемо и прозрачно;
* позволять внешней системе или оператору контролировать степень вмешательства;
* обеспечивать качество результата через автоматическую и полуавтоматическую оценку.

# 4. Обязательные проектные принципы

## 4.1. Система должна разделять смысл и стиль

Архитектура должна быть выстроена так, чтобы:

* **смысл** рассматривался как инвариантная сущность;
* **стиль** рассматривался как управляемая переменная.

Это означает, что все ключевые модули должны различать:

* смысловое содержимое текста;
* выразительные свойства текста.

## 4.2. Система должна быть reference-driven

В основе системы должны лежать **Style Libraries** — библиотеки референсных текстов, из которых система строит профиль стиля.

Научное обоснование: исследование HART (Bao et al., 2025) устанавливает, что подход **имитации (mimicking)** — то есть переписывание с использованием конкретного референсного текста — создаёт наиболее существенный сдвиг в распределении текстовых признаков по сравнению с другими методами гуманизации. Ссылка: `docs/research/kimi_arxiv_humanization_overview.md`.

## 4.3. Система должна быть guided, а не blind

Переписывание должно быть устроено не по схеме "переписали, потом проверили", а по схеме:

* генерация нескольких вариантов;
* оценка вариантов внешними модулями;
* выбор лучшего или нескольких лучших;
* финальная полировка;
* оценка.

То есть rewrite engine должен работать совместно с **guidance-логикой**. Это соответствует ранее зафиксированной архитектуре проекта и идее guided rewriting, а не слепого rewrite с одной финальной проверкой.

**Научное обоснование расширенной guided-логики:** работа `docs/research/2506.07001_adversarial_paraphrasing.pdf` (Cheng et al., 2025) демонстрирует, что управление генерацией на уровне отдельных токенов (adversarial paraphrasing) достигает среднего снижения T@1%F на **87.88%** против 30.27% при обычном парафразировании. Хотя задача той работы — обход детекторов — иная, принцип **guidance signal на каждом шаге генерации** напрямую применим к задаче style-guided rewriting.

## 4.4. Система должна быть модульной

Каждая крупная функция должна быть реализована отдельным модулем или сервисным слоем с чётким контрактом.

## 4.5. Система должна быть пригодна для human-in-the-loop сценариев

Даже если первичная интеграция будет машинной, система должна возвращать достаточно данных для:

* визуального diff;
* объяснения решений;
* выбора варианта;
* ручной редакторской корректировки.

## 4.6. *(Новый принцип)* Система должна управлять burstiness и perplexity

На основании анализа `docs/research/kimi_humanize_ai_pro_analysis.md` и `docs/research/2404.01907_hmgc_adversarial_framework.pdf`:

ИИ-модели генерируют текст со **статистически сглаженной равномерностью** — предсказуемым ритмом и низкой perplexity. Система должна уметь целенаправленно:

* варьировать длину предложений (burstiness);
* контролировать лексическую предсказуемость (perplexity management);
* чередовать сложные и простые конструкции.

Это должно быть **управляемым параметром** стилевого профиля, а не случайным эффектом.

# 6. Основные сущности доменной модели

## 6.1. Style Library

Сущность, представляющая библиотеку референсных текстов одного типа.

Содержит:

* имя библиотеки;
* описание;
* категорию;
* язык;
* область применения;
* владельца;
* проект;
* статус;
* версию;
* список текстов-образцов;
* рассчитанный стиль-профиль;
* уровень качества библиотеки.

## 6.2. Style Sample

Отдельный текст внутри библиотеки.

Содержит:

* текст;
* заголовок;
* источник;
* тип контента;
* предполагаемый стиль;
* автор;
* язык;
* качество;
* служебные метаданные;
* результат анализа.

## 6.3. Style Profile

Вычисленный стилевой профиль библиотеки.

Содержит:

* формальность;
* среднюю длину предложений;
* разброс длины;
* **burstiness index** *(новое поле)* — коэффициент вариативности длин предложений;
* **target perplexity range** *(новое поле)* — целевой диапазон перплексии;
* ритм;
* типичную плотность абзацев;
* характер переходов;
* степень образности;
* частотные маркеры;
* нежелательные маркеры;
* типовые синтаксические паттерны;
* профиль композиции;
* **11 лингвистических маркеров** *(новое поле)*: индекс читаемости, средняя длина слова, лексическое разнообразие, частота сокращений, паттерны пунктуации и др. — согласно таксономии из `docs/research/kimi_arxiv_humanization_overview.md`;
* служебные признаки для style guidance.

## 6.4. Semantic Contract

Сущность, описывающая смысловые ограничения rewrite-задачи.

Содержит:

* защищённые сущности;
* защищённые факты;
* числовые ограничения;
* причинно-следственные опоры;
* список терминов с обязательным сохранением;
* список чувствительных сегментов;
* allowed rewrite level;
* режим strict/balanced/loose;
* **maximum perturbed ratio** *(новое поле)* — максимальная доля изменённых токенов (MPR constraint);
* **USE similarity threshold** *(новое поле)* — порог семантической близости через Universal Sentence Encoder;
* **POS constraint flag** *(новое поле)* — флаг обязательного соблюдения частей речи при заменах.

## 6.5. Rewrite Task

Конкретная задача переписывания.

Содержит:

* исходный текст;
* выбранную библиотеку стиля;
* режим rewrite;
* режим semantic contract;
* входные ограничения;
* идентификатор проекта;
* идентификатор пользователя / клиента;
* служебное состояние выполнения.

## 6.6. Rewrite Variant

Один вариант результата.

Содержит:

* итоговый текст;
* промежуточные показатели;
* метрики;
* оценку style match;
* оценку semantic preservation;
* **perplexity score** *(новое поле)*;
* **burstiness score** *(новое поле)*;
* **fluency win rate** *(новое поле)* — для pairwise оценки;
* объяснение изменений;
* статус пригодности.

## 6.7. Evaluation Report

Итоговый отчёт оценки качества.

Содержит:

* абсолютные метрики;
* judge-оценки;
* pairwise-оценки (fluency win rate);
* нарушения semantic contract;
* предупреждения;
* рекомендации.

## 6.8. Style Preset

Сохранённая конфигурация работы системы.

Содержит:

* Style Library;
* rewrite mode;
* semantic contract mode;
* набор ограничений;
* уровень вмешательства;
* активные эвристики.

# 7. Описание модулей

## 7.1. Reference Library Manager

### Назначение

Управление библиотеками референсных текстов.

### Обязанности

* создание новой библиотеки;
* добавление текстов;
* импорт текстов пакетами;
* обновление библиотеки;
* связывание библиотеки с проектом;
* версионирование;
* архивирование;
* удаление / деактивация;
* переиндексация.

### Требования

* библиотека должна поддерживать тексты одного языка;
* библиотека должна иметь категорию;
* библиотека должна поддерживать служебные теги;
* должна быть возможность хранить несколько библиотек для одного проекта;
* должна быть возможность создавать как "личные", так и "проектные" библиотеки.

### Важное правило

Библиотека не должна считаться валидной для профилирования, пока не пройдет минимальную проверку качества и объёма.

---

## 7.2. Corpus Quality Tiering

### Назначение

Оценка качества библиотеки и отдельных sample'ов.

### Почему это обязательно

Корпус из смешанных или слабых текстов создаёт плохой стилевой профиль. Следовательно, система должна определять качество корпусов, а не принимать все тексты как равнозначные.

### Уровни качества

* **L1** — сильный, чистый, консистентный стиль: сохраняет тон, уровень словарного запаса, сложность и соответствует источнику.
* **L2** — рабочий, но неоднородный стиль: сохраняет смысл, но снижает общее качество, упрощает лексику или структуру.
* **L3** — шумный, слабый или конфликтный материал: добавляет бессмысленные фразы, конструирует некорректные предложения, искажает смысл.

### Научное обоснование трёхуровневой классификации

Классификация L1/L2/L3 опирается на методологию из `docs/research/2501.03437_damage_humanizer_detection.pdf` (Masrour et al., 2025). Исследование доказывает, что **включение в обучающий набор только L1 данных** сохраняет низкий false positive rate, тогда как включение L2/L3 данных значимо повышает его. Та же логика применима к качеству стилевого профиля: обучение на смешанном корпусе L1+L2+L3 даёт нестабильный профиль.

Метрика **Fluency Win Rate** (предложена в той же работе) — доля случаев, когда обработанный текст оценивается как более связный, чем оригинал, — может быть адаптирована для автоматической оценки tier sample'а.

### Что модуль должен уметь

* оценивать каждый загруженный sample;
* оценивать библиотеку целиком;
* определять, можно ли использовать sample в основном профиле;
* определять, нужно ли исключить sample;
* формировать рекомендации по очистке библиотеки;
* вычислять **Fluency Win Rate** для пачки sample'ов через LLM-судью;
* обнаруживать hallucinated citations и nonsensical insertions (признаки L3).

### Практическая логика

* в расчёт основного Style Profile по умолчанию должны входить только L1 и, при необходимости, L2 sample'ы;
* L3 sample'ы должны либо исключаться, либо учитываться только в исследовательском/вспомогательном контуре.

## 7.3. Style Profile Engine

### Назначение

Построение машиночитаемого профиля стиля по библиотеке.

### Что должен анализировать

* длину предложений и распределение длин;
* **burstiness** — коэффициент вариативности длин предложений (CV = std/mean);
* длину абзацев;
* ритм;
* уровень формальности;
* плотность оценочной лексики;
* типичные переходы;
* уровень образности;
* частоту определённых вводных конструкций;
* характер синтаксиса;
* композиционные привычки;
* предпочтительные и нежелательные лексические маркеры;
* **11 лингвистических маркеров человечности** (по классификации из `docs/research/kimi_arxiv_humanization_overview.md`):
  - использование сокращений;
  - индексы читаемости (Flesch, Dale-Chall);
  - средняя длина слова;
  - вариативность длины предложений;
  - лексическое разнообразие (TTR, MATTR);
  - частотные паттерны пунктуации;
  - доля существительных, глаголов, прилагательных;
  - доля фразовых глаголов и идиом;
  - доля сложносочинённых и сложноподчинённых предложений;
  - средняя глубина синтаксического дерева;
  - частота личных местоимений и вопросительных конструкций.

### Выходные данные

Модуль должен выдавать:

1. **описательный профиль стиля**;
2. **сигналы для guidance-модуля**;
3. **признаки для style match scoring**;
4. **сигналы отклонения от стиля**;
5. **целевой диапазон perplexity** — вычисляется по корпусу и используется как ориентир для rewrite engine.

### Важное требование

Этот модуль не должен ограничиваться статистическим summary. Он должен строить именно **Style DNA**, пригодный для дальнейшего управления rewrite-задачей.

## 7.4. Input Analyzer

### Назначение

Анализировать входной текст до переписывания.

### Обязанности

* определять жанровые характеристики;
* оценивать текущую формальность;
* выявлять шаблонность;
* оценивать плотность фактов;
* оценивать насыщенность конкретикой;
* находить повторяемые паттерны;
* строить risk map;
* рекомендовать глубину rewrite;
* **оценивать текущую perplexity** входного текста *(новое)*;
* **оценивать текущий burstiness** входного текста *(новое)*.

### Выход

* входной технический профиль;
* карта участков с высоким риском потери смысла;
* карта участков с высокой стилевой гибкостью;
* **perplexity gap** — разница между текущей perplexity и целевым диапазоном из Style Profile.

## 7.5. Semantic Contract Builder

### Назначение

Построить контракт, защищающий смысл текста.

### Обязанности

* извлечь ключевые сущности;
* извлечь числа, даты, имена, термины;
* определить чувствительные фрагменты;
* определить обязательные для сохранения утверждения;
* определить допустимую глубину rewrite.

### Режимы

* **Strict** — максимальная защита смысла, минимальные изменения.
* **Balanced** — сбалансированный режим, большинство изменений допустимы.
* **Loose** — разрешены глубокие изменения, только имена/числа защищены.

### Внутренняя карта важности

Модуль должен разделять содержимое текста на:

* must-preserve;
* high-risk;
* replaceable;
* style-flex.

### Параметры Rewrite Constraint

Semantic Contract Builder должен также формировать параметры для **Rewrite Constraint Layer** (см. 7.10):

* **maximum perturbed ratio (MPR)** — максимальная доля заменённых токенов (напр., не более 40% от исходного текста). Научная основа: `docs/research/2404.01907_hmgc_adversarial_framework.pdf`.
* **POS constraint list** — список POS-меток, замена которых запрещена или ограничена.
* **USE similarity window** — размер контекстного окна для проверки семантической близости через Universal Sentence Encoder.

### Дополнительное требование

Semantic Contract Builder должен уметь возвращать объяснимые ограничения, пригодные для внешнего API и пользовательского интерфейса.

## 7.6. Guided Rewrite Engine

### Назначение

Выполнять управляемое переписывание текста.

### Требования к реализации

* базироваться на encoder-decoder логике rewrite;
* быть совместимым с внешними LLM через абстракции;
* поддерживать несколько режимов генерации;
* уметь работать по сегментам;
* поддерживать генерацию нескольких вариантов;
* не быть привязанным к одному поставщику LLM.

### Обязательные режимы

* **Conservative** — минимальные изменения, сохранение структуры.
* **Balanced** — умеренная трансформация.
* **Expressive** — глубокая трансформация под целевой стиль.

### Обязательные требования

* учёт semantic contract;
* учёт style profile;
* возможность отдачи нескольких кандидатов на следующий этап;
* отсутствие жёсткой зависимости от OpenAI API;
* возможность использовать OpenAI API как один из провайдеров.

### Prompt-инжиниринг для guided rewrite *(новый раздел)*

Система должна использовать два базовых шаблона промпта, обоснованных исследованием `docs/research/kimi_arxiv_humanization_overview.md`:

**Режим Diversifying:**
```
Revise the text to enrich its language diversity, employing varied sentence
structures, synonyms, and stylistic nuances, while preserving the original
meaning: {input_text}
```

**Режим Mimicking (reference-driven):**
```
Rewrite the text using the same language style, tone, and expression as the
reference text. Focus on capturing the unique vocabulary, sentence structure,
and stylistic elements evident in the reference:
{input_text}

# Reference Text:
{reference_sample}
```

В режиме Mimicking в качестве `reference_sample` используется 1–3 репрезентативных текста из Style Library (L1-класса).

Оба шаблона являются **базовыми** — агент должен строить поверх них промпты с учётом параметров semantic contract и style profile.

## 7.8. Style Guidance Engine

### Назначение

Направлять rewrite-процесс к целевому стилю.

### Что должен делать

* получать несколько candidate rewrites;
* оценивать их по совпадению со стилем;
* учитывать semantic safety;
* учитывать ритм;
* учитывать композицию;
* штрафовать варианты, уходящие в плохую гладкость, штампы или потерю авторского профиля;
* отдавать ranked list кандидатов.

### Расширенная guided-логика на уровне токенов *(новый раздел)*

На основании `docs/research/2506.07001_adversarial_paraphrasing.pdf` (Adversarial Paraphrasing framework):

Для режима **Precision Guided** Style Guidance Engine должен поддерживать пошаговый guidance при генерации:

1. Paraphraser LLM генерирует распределение вероятностей на следующий токен.
2. Применяется **top-k** и **top-p** фильтрация (рекомендуемые значения: k=50, p=0.99).
3. Для каждого кандидата-токена вычисляется **style score** — степень соответствия целевому стилевому профилю.
4. Выбирается токен с максимальным style score (или минимальным отклонением от Style DNA).
5. Цикл продолжается до генерации EOS.

Этот режим **опциональный** — он требует интеграции с совместимым inference backend — но архитектура должна предусматривать его подключение без переработки core pipeline.

**Научное обоснование:** работа показывает, что token-level guidance в сравнении с post-hoc ranking даёт кратно лучший результат (среднее снижение T@1%F 87.88% против 30.27%). Принцип напрямую переносится в задачу style alignment.

### Важно

Этот модуль должен работать **не только после генерации**, но и как рабочий управляющий слой в rewrite pipeline.

## 7.9. Holistic Lexical Substitution Ranker

### Назначение

Управлять локальными заменами слов и фраз с учётом влияния замены на **весь текст целиком**, а не только на целевую позицию.

### Научное обоснование

Модуль опирается на результаты `docs/research/2509.11513_holistic_lexical_substitution.pdf` (Hu et al., 2025). Ключевой вывод: при ранжировании кандидатов для лексической замены учёт **holistic sentence semantics** (косинусное сходство между оригиналом и заменённой версией по всем токенам) значимо превосходит подход, учитывающий только сходство целевого токена.

Формальная модель (Score formula из статьи):

```
Score(s, s') = Σ_i w_i · cos(f(x_i), f(x'_i))

где:
  f(x_i) = concat(h^l_i) для l ∈ [start, end]  — конкатенация представлений по слоям
  w_i     — веса влияния токенов на целевой токен
```

Веса вычисляются двумя способами (оба должны быть реализованы):

**Метод 1: Attention-based weights**
- Усредняются multi-head attention веса по слоям и головам.
- Из внимания извлекается вектор влияния каждого токена на целевой.
- Нормализуется через softmax (исключая сам целевой токен).
- Вес целевого токена фиксируется = 1.

**Метод 2: Integrated Gradients weights (рекомендуемый)**
- Вычисляется интегральный градиент по пути от baseline (pad-токен) к реальному токену.
- Более интерпретируемый и теоретически более надёжный, чем attention.
- Вычислительно дороже, но даёт лучшие результаты (GAP на SWORDS: 64.4 vs 61.3 у BERT).
- Для embedding-моделей IG-цель определяется как L2-норма вектора целевого токена.

Рекомендуемые модели для Ranker: **DeBERTa-v3-large** (наивысшие GAP-метрики в обоих датасетах); также доступны MPNet-v2, BERT-large, SimCSE.

### Основные требования

* оценивать замену не только по слову, но и по влиянию на весь сегмент;
* учитывать контекст через weighted cosine similarity;
* учитывать style profile;
* учитывать semantic contract;
* учитывать phrase-level coherence;
* использовать представления из нескольких слоёв (рекомендуемый диапазон: слои 3 — (N-2), где N — общее число слоёв).

### Режимы

* **Fast mode** — attention-based weights, быстрее, подходит для real-time UI.
* **Precision mode** — Integrated Gradients weights, точнее, подходит для batch-обработки.

### Где должен применяться

* точечная пост-обработка;
* финальный polishing;
* интерактивные редакторские предложения;
* локальные rewrite-операции.

## 7.10. Rewrite Constraint Layer *(новый обязательный модуль)*

### Назначение

Контролировать допустимость каждой локальной замены или сегментного изменения с точки зрения качества и смысловой целостности.

### Научное обоснование

Модуль основан на системе ограничений из `docs/research/2404.01907_hmgc_adversarial_framework.pdf` (HMGC framework, Zhou et al., 2024). В той работе ограничения предотвращают семантический дрейф при adversarial word substitution. В нашей системе та же логика применяется для гарантии корректности lexical substitution и structural rewrite.

### Три обязательных ограничения

#### Constraint 1: POS (Part-of-Speech) Constraint

* Заменяющее слово должно принадлежать той же части речи, что и заменяемое.
* Прилагательные не заменяют существительные. Глаголы не заменяют наречия.
* Реализация: spaCy или аналог для POS-тегирования.
* Исключения: конфигурируются в semantic contract (например, допустима замена существительного на герундий).

#### Constraint 2: Maximum Perturbed Ratio (MPR)

* Суммарная доля заменённых токенов не должна превышать заданный порог.
* Рекомендуемый дефолт: **40%** токенов исходного текста.
* Для режима Conservative: 20%, Balanced: 40%, Expressive: 60%.
* MPR рассчитывается на уровне предложения и на уровне текста.

#### Constraint 3: USE (Universal Sentence Encoder) Similarity Constraint

* После каждой замены вычисляется косинусное сходство между оригинальным контекстным окном и изменённым.
* Размер окна: рекомендуется 50 токенов.
* Минимальный порог схожести: **0.75** (конфигурируемо).
* Если порог нарушен — замена отклоняется.
* Реализация: Universal Sentence Encoder или аналог (paraphrase-mpnet-base-v2).

### Дополнительные ограничения

* **Protected spans constraint**: токены из must-preserve зон semantic contract не могут быть изменены.
* **Sentence drift constraint**: максимальное структурное отклонение предложения от оригинала.
* **Paragraph drift constraint**: контроль дрейфа на уровне абзаца.

### Место в pipeline

Rewrite Constraint Layer работает как **фильтр** между генерацией кандидатов и их применением — как в Holistic Lexical Substitution Ranker, так и в Rule-Based Structural Polishing.

## 7.11. Word Importance Scorer *(новый обязательный модуль)*

### Назначение

Определять, какие слова в тексте наиболее важны для изменения с точки зрения стилевой трансформации и наиболее безопасны с точки зрения смысла.

### Научное обоснование

Модуль основан на dual-aspect word importance ranking из `docs/research/2404.01907_hmgc_adversarial_framework.pdf`. Ключевое открытие: при выборе слов для замены наиболее эффективна **комбинация градиентной значимости и значимости по perplexity**.

### Два компонента важности

#### Компонент 1: Gradient-based importance

```
I^g_wi = ||∂L/∂e_wi||_1

где e_wi — embedding-вектор i-го токена
    L    — функция потерь суррогатной модели
```

Токены с высоким градиентом имеют наибольшее влияние на предсказание модели → их изменение наиболее эффективно для стилевой трансформации.

#### Компонент 2: Perplexity-based importance

```
I^p_wi = ppl(t \ w_i) − ppl(t)

где t \ w_i — текст без i-го токена
```

Токены, удаление которых сильно меняет perplexity, — структурно значимые для стиля. Их изменение при rewrite даёт наибольший эффект на воспринимаемую "ИИ-ность" или "человечность" текста.

#### Итоговая формула

```
I_wi = (1 − α) · I^g_wi + α · I^p_wi

где α — параметр баланса (рекомендуемое значение: 0.2)
```

### Применение в системе

* **Holistic Lexical Substitution Ranker** использует Word Importance Scorer для приоритизации кандидатов замены.
* **Guided Rewrite Engine** использует Word Importance Scorer для определения, какие сегменты требуют наибольшей трансформации.
* **Rule-Based Structural Polishing** использует Word Importance Scorer для точечного polishing наиболее "стилистически уязвимых" мест.

### Реализация

* суррогатная модель для gradient computation: RoBERTa-based классификатор, обученный на парах (оригинал / rewrite);
* perplexity вычисляется через LLM (предпочтительно: небольшая open-source модель типа GPT-2 medium или Pythia для эффективности).

## 7.14. Evaluation Engine

### Назначение

Проводить полноценную оценку результата.

### Три режима оценки

#### 1. Absolute Metrics

**Семантические:**
* **BERTScore F1** — целевое значение > 0.90 (обоснование: `docs/research/kimi_arxiv_humanization_overview.md`);
* entity preservation;
* fact preservation.

**Стилевые:**
* style match — сходство с Style DNA библиотеки;
* **Linguistic Marker Shift** — направленный сдвиг по 11 маркерам от исходного профиля к целевому.

**Читаемость и разнообразие:**
* readability (Flesch, Dale-Chall);
* lexical diversity (TTR, MATTR);
* repetition score;
* cliché score.

**Perplexity и burstiness:**
* **perplexity score** — целевой диапазон берётся из Style Profile; ориентир: человеческие тексты ~15 PPL (LLaMA-3.1-8B); AI-тексты ~14.9 PPL; простой парафраз ~9.3 PPL (из `docs/research/2506.07001_adversarial_paraphrasing.pdf`);
* **burstiness score** — коэффициент вариации длин предложений;
* **chrF++** — целевое значение > 50.

**Дополнительные метрики (опциональные для MVP):**
* ROUGE-L — лексическое перекрытие;
* Δflesch% — изменение читаемости после rewrite.

#### 2. Judge Evaluation

* оценка LLM-судьёй (рекомендуется GPT-4o или эквивалент);
* проверка естественности;
* проверка сохранения смысла;
* проверка стилевого соответствия;
* **Likert-scale качества (1–5)** — по методологии из `docs/research/2506.07001_adversarial_paraphrasing.pdf`:
  - 5: эквивалентен оригиналу, различия только в формулировках;
  - 4: почти эквивалентен, незначительные отличия;
  - 3: частично эквивалентен, некоторые важные детали изменились;
  - 2: тематически связан, но большая часть смысла потеряна;
  - 1: нет тематической связи.

#### 3. Pairwise Comparison

* сравнение вариантов друг с другом;
* выбор лучшего варианта по смыслу, стилю, ясности и естественности;
* **Fluency Win Rate** *(новое)* — адаптация метрики из `docs/research/2501.03437_damage_humanizer_detection.pdf`:
  - LLM-судья (GPT-4o) выбирает наиболее fluent и coherent из двух вариантов;
  - возвращаются: wins / ties / losses для каждого кандидата;
  - рекомендуемый размер выборки для оценки: 25 sample'ов.

### Результат работы модуля

* структурированный evaluation report;
* итоговый composite score;
* объяснение слабых мест;
* предупреждения;
* **рекомендации по улучшению** (перплексия слишком низкая → увеличить burstiness, семантика нарушена → применить strict semantic contract и т.д.).

# 9. Интеграционный сценарий

Система должна уметь работать так:

1. Внешний проект создаёт Style Library.
2. Загружает 10–50 референсных текстов.
3. Система рассчитывает quality tiering (L1/L2/L3) и style profile (включая burstiness index и target perplexity range).
4. Внешний проект отправляет исходный текст на rewrite.
5. **Input Analyzer** определяет perplexity gap и строит risk map.
6. Система строит semantic contract (включая MPR, POS constraints, USE threshold).
7. **Word Importance Scorer** вычисляет карту приоритетов замен.
8. **Guided Rewrite Engine** генерирует несколько rewrite-вариантов.
9. **Style Guidance Engine** оценивает и ранжирует варианты.
10. **Holistic Lexical Substitution Ranker** проводит точечную коррекцию лексики.
11. **Rewrite Constraint Layer** верифицирует все замены (POS, MPR, USE).
12. **Rule-Based Structural Polishing** корректирует burstiness и убирает артефакты.
13. **Evaluation Engine** вычисляет финальные метрики (BERTScore, perplexity, burstiness, fluency win rate, Likert-score).
14. Внешний проект получает:
    * итоговые тексты;
    * scores (composite + breakdown);
    * diff / explanation data;
    * warnings;
    * word importance map.

# 10. Требования к объёму корпусов стиля

Агент-разработчик должен учитывать в логике системы:

* библиотека из 5–7 текстов может использоваться только как слабый старт;
* библиотека из 10–15 текстов — минимально рабочая;
* библиотека из 20–30 текстов — хороший рабочий объём;
* библиотека из 30–50 текстов — сильный корпус при условии однородности.

**Дополнительное требование из исследований:** референсный текст в режиме Mimicking должен составлять **200–500 слов** для оптимального стилевого трансфера. Меньший объём — нестабильные паттерны, больший — диффузия внимания. Источник: `docs/research/kimi_arxiv_humanization_overview.md`.

Система должна диагностировать не только объём, но и **качество и однородность** библиотеки.

# 11. Логика работы с авторами и смешанными корпусами

Система должна поддерживать два режима библиотек:

## 11.1. Single-voice library

Для конкретного авторского голоса.

## 11.2. Editorial-style library

Для стилевого профиля жанра / рубрики / медиа / бренда.

Система должна различать:

* библиотеку "одного голоса";
* библиотеку "общего редакционного стиля".

* либо сигнализировать о неоднородности;

# 17. Что не должно быть упущено разработчиком

Агент обязан учесть:

1. что система — **API-модуль**, а не привязанный UI-продукт;
2. что OpenAI API должен быть встроен как внешний LLM-провайдер;
3. что архитектура должна оставаться provider-agnostic;
4. что смысл должен быть защищён через semantic contract;
5. что rewrite должен быть guided;
6. что style profile должен быть пригоден не только для хранения, но и для управления rewrite;
7. что библиотеки стиля должны проходить quality tiering (L1/L2/L3);
8. что оценка результата должна быть многослойной;
9. что всё должно быть Python-friendly;
10. что кодовые детали агент выбирает сам, но архитектурные принципы менять нельзя;
11. **что lexical substitution должна учитывать влияние замены на весь текст, а не только на позицию слова** (Holistic Lexical Substitution Ranker с attention/IG weights);
12. **что качество корпуса определяется уровнем L1/L2/L3, и только L1-образцы должны участвовать в построении основного Style Profile**;
13. **что rewrite pipeline должен соблюдать три ограничения при каждой замене: POS, MPR, USE** (Rewrite Constraint Layer);
14. **что perplexity и burstiness — измеримые параметры стиля**, которые должны входить в Style Profile и вычисляться для каждого результата;
15. **что Fluency Win Rate — обязательная pairwise метрика** в Evaluation Engine, не опциональная.

# 18. Итоговая формулировка задачи для агента

Разработать **модульную backend-систему авторского переписывания текста с опорой на пользовательские библиотеки референсных текстов**, реализованную на Python-стеке, подключаемую по API к любому внешнему проекту, с поддержкой:

* управления стилевыми библиотеками с quality tiering (L1/L2/L3),
* построения Style Profile (включая burstiness, perplexity target, 11 лингвистических маркеров),
* защиты смысла через Semantic Contract с POS/MPR/USE constraints,
* guided rewrite pipeline с Word Importance Scoring,
* Holistic Lexical Substitution с attention-based и Integrated Gradients weights,
* внешних LLM через OpenAI API и расширяемую provider abstraction,
* многослойной оценки качества (BERTScore, perplexity, burstiness, Likert-scale, Fluency Win Rate),
* human-in-the-loop совместимости,
* PostgreSQL + Redis + FastAPI + async jobs архитектуры,
* полной внутренней документации в `docs/`.
